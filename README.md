# Лисий диспетчер

![Лисий диспетчер анализирует сигналы лесного радара](docs/fox-dispatcher-cover.png)

Отчёт по сценарию «Лисий диспетчер» для тестового задания MOX на позицию AI-first Developer.
Приложение группирует наблюдения по `fox_id`, рассчитывает баллы, строит рейтинг и объясняет, почему конкретная лиса оказалась лидером.

- [Открыть готовую версию](https://396b5341.ht-ml.app/)
- Открыть автономную сборку из файла `dist/index.html`

## Как запустить

### Самый простой способ

Посмотреть сайт можно по адресу [https://396b5341.ht-ml.app/](https://396b5341.ht-ml.app/).
Ничего устанавливать или вводить в терминале для этого не потребуется.

### Если готовая версия не открывается

Скачайте проект со страницы на GitHub:

1. Нажмите зелёную кнопку `Code`.
2. Выберите `Download ZIP`.
3. Распакуйте архив.
4. Перейдите в папку `dist` и дважды щёлкните по файлу `index.html`.

Сайт запустится в браузере.

<details>
<summary>Локальный запуск из исходного кода</summary>

Для запуска из исходного кода понадобится Node.js 20.19+ из ветки 20.x или Node.js 22.12+.
Откройте папку проекта в терминале и выполните следующие команды:

```bash
npm install
npm run dev
```

Терминал покажет адрес приложения, обычно [http://localhost:5173/](http://localhost:5173/).
Откройте его в браузере и не закрывайте терминал с Vite, пока работает сервер.
Для остановки нажмите `Ctrl+C`.

Команда `npm run check` запускает ESLint, пять модульных тестов и сборку приложения.

</details>

## Выбранный сценарий

Для проекта я выбрал сценарий «Лисий диспетчер».
В интерфейсе можно редактировать наблюдения и менять вес каждого сигнала.
После этого рейтинг пересчитывается: приложение показывает, какая лиса выглядит наиболее подозрительно, и объясняет, почему.
Во вкладке AI Worklog отражены семь ключевых этапов работы с нейросетью.

## AI-инструменты и проверка

Codex помог мне разобрать требования, сравнить варианты скоринга, подготовить прототип, написать тесты и провести ревью.
Сложнее всего было согласовать с агентом общую логику приложения.
Больше всего времени заняли обсуждение функционала, верстки и сравнение вариантов прототипа.

Когда PRD был готов, я нарезал через скилл спецификацию на задачи и передал их на реализацию.
Codex с помощью Playwright проверил интерфейс в десктопном и мобильном адаптиве, а я дополнительно вручную проверил верстку и функционал на 1920 адаптиве.

Опыт QA инженера помог подсветить агенту проблемы в вёрстке и функционале: неверные размеры полей ввода, смещение значений и резкие скачки страницы при добавлении и удалении записей.
Очень помог Lavish - через него удобно и быстро передавать правки по верстке агенту.
